Meaningful reward function with partial progress signals
Baseline inference script with reproducible scores
Deploy to Hugging Face Spaces + working Dockerfile
README with environment description, action/observation spaces, setup instructions

Problem statement:
Focus on Triage & Efficiency Build an autonomous Search and Rescue agent to locate and save people in simulated earthquake ruins. Using a standardized legged robot, the agent must use smart triage to save at risk victims first. The challenge is to optimize sensor use and pathfinding to stay within a strict 90-120 minute power budget while navigating through debris.

SCORING SYSTEM 

1. Introduction
The scoring system assesses performance across five core dimensions:

Operational Safety
Victim Handling
Decision Intelligence
Operational Efficiency
Time Performance

The system ensures non redundant evaluation metrics, real world alignment with rescue priorities, robust penalty mechanisms, and fair scoring across all difficulty levels.

2. Final Reward Formula: 
S_final =0.20 * S_safety +0.30 * S_victim +0.20 * S_decision +0.20 * S_efficiency +0.10 * S_time

Total Weight = 1.00

3. Absolute Penalty System

Penalties are applied after computing the final score:

S_final_adjusted = max(0.0, S_final - sum_of_penalties)

Penalty Definitions:

Crush injury extracted without prior flag: -0.25
Robot entered critical gas zone: -0.15
False explosion triggered: -0.30
Robot destroyed due to preventable cause: -0.20

Final score is clipped to a minimum of 0.0.

4. Pillar 1: Operational Safety (Weight = 0.20)

4.1 S_survival (Weight = 0.10)

Range: 0.0 to 1.0
Base Value: 1.0

Collision Definition:

Minor debris contact is not considered a collision.
Impact greater than 0.3 m/s is considered a collision.
Movement causing structural disturbance is considered a collision.

Penalty Structure:

First collision: -0.04
Second collision: -0.08
Third collision: -0.15
Collision near survivor: -0.25

Additional Penalties:

Joint damage: -0.08
Tip over with recovery: -0.10

Reasoning:
Ensures safe navigation and minimizes additional damage in fragile environments.

4.2 S_stability (Weight = 0.10)

S_stability =(1 - (instability_events / total_steps)) * 0.7 +(stability_recovery * 0.3)

Recovery Component:

If no instability events: stability_recovery = 1.0
Otherwise: stability_recovery = successful_recoveries / total_events

Reasoning:
Evaluates both terrain handling and recovery capability.



5. Pillar 2: Victim Handling (Weight = 0.30)

5.1 Detection Accuracy (Weight = 0.08)

S_detection =(TP / total_victims) * 0.7 +(confidence * 0.3) -(FP * 0.15)

Definitions:

True Positive (TP): Victim detected within 2 meters
False Positive (FP): Incorrect detection

Reasoning:
Measures sensing reliability independently from exploration behavior.

5.2 S_victim_location (Weight = 0.10)

S_location =(accurately_located / detected) * 0.6 +(precision * 0.4)

Precision Threshold:

Error ≤ 0.5 m → Score 1.0
Error ≤ 1.5 m → Score 0.7
Error ≤ 3.0 m → Score 0.3
Error > 3.0 m → Score 0.0

The unit “m” refers to meters, which measure the distance error between the robot’s predicted victim location and the actual victim position. This threshold defines how accurately the robot can localize a victim. Smaller errors indicate higher precision and receive higher scores, while larger errors reduce the score. This ensures that the robot is rewarded not just for detecting victims, but for identifying their exact location accurately, which is critical for efficient rescue operations.

Reasoning:
Accurate localization is essential for effective rescue.

5.3 Rescue Score (Weight = 0.10)

S_rescue =(rescued / detected) * 0.6 +(successful_rescues / attempts) * 0.4

Penalties:

Improper handling: -0.10
Dropped victim: -0.15

Reasoning:
Evaluates execution effectiveness during rescue operations.

5.4 Rescue Order (Weight = 0.08)

S_order =(correct_priority_rescues / total_rescues)

Penalty:
Incorrect rescue order (low priority before critical): -0.10 per event

Reasoning:
Ensures prioritization aligns with survival urgency.

6. Pillar 3: Decision Intelligence (Weight = 0.20)

6.1 Priority Score (Weight = 0.12)

S_priority =(correct_priority_assignments / total_victims)

Considerations:

Health condition
Accessibility
Survival probability

Penalty:
Misclassification of a critical victim: -0.15

Reasoning:
Represents the core decision-making capability of the system.

6.2 Environment Scanning (Weight = 0.08)

S_env =(useful_insights / total_scans) * 0.6 +(coverage * 0.4)

Penalties:
Missed major hazard: -0.10
Redundant scanning: -0.05

Reasoning:
Captures environmental awareness while avoiding redundancy with sensor metrics.

7. Pillar 4: Operational Efficiency (Weight = 0.20)

7.1 S_energy (Weight = 0.10)

S_energy =(remaining_battery / 100) * 0.5 +(work_accomplished / energy_used) * 0.5

Work Definition:

work =(2 * victims) +(1 * hazards) +(0.5 * zones)-(0.3 * unnecessary_actions)

Reasoning:
Encourages productive energy utilization.

7.2 S_path_efficiency (Weight = 0.10)

S_path = 1 - (revisit_ratio * 0.4) - (idle_ratio * 0.3) + (smoothness * 0.3)

Reasoning:
Measures practical navigation efficiency without relying on ideal path assumptions.

8. Pillar 5: Time Performance (Weight = 0.10)

8.1 S_time

S_time =max(0, 1.0 if within limit,otherwise (time_remaining / limit) * 0.6 + 0.4)

Golden Hour Limits:

Easy: 90 minutes
Medium: 60 minutes
Hard: 45 minutes

Reasoning:
Ensures urgency without producing negative scores.

9. System Design Advantages

Non-Redundancy:

Each metric evaluates a distinct capability. Detection, localization, decision making, and execution are all independent.

Robustness:
Absolute penalties prevent unsafe exploitation.
Score flooring ensures stability.

Scalability:
The same scoring system applies consistently across multiple environments and all difficulty levels.

10. Conclusion:
This scoring system provides a balanced, realistic, and implementation-ready evaluation model for autonomous rescue systems. It emphasizes safety, intelligent decision making, and efficient execution while ensuring fairness and consistency across varying scenarios.
